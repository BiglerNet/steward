## Context

`docker-compose.yml` today only runs Postgres; the API and frontend run natively on the host via `dotnet run`/`npm run dev`. The end deployment target is self-hosted Kubernetes via a Helm chart, with the API's document storage (`registration-and-warranty`) expecting an NFS-backed `PersistentVolumeClaim` mounted at `Storage:RootPath`. Building the Compose stack now — with the same service/volume/env-var shape a Helm chart would have — means the Dockerfiles, volume mount, and externalized configuration get debugged locally on a fast iteration loop instead of discovered for the first time against a k8s cluster.

## Goals / Non-Goals

**Goals:**
- Dockerize the API and the frontend as production-style images (not dev-server containers).
- Add `api` and `web` services to `docker-compose.yml`, networked with the existing `postgres` service.
- Mount a local volume for `Storage:RootPath` into the `api` container, mirroring the eventual NFS-backed PVC.
- Externalize all configuration that would later be Helm `values.yaml`/`Secret`/`ConfigMap` entries as Compose environment variables now, via a `.env` file pattern.
- Shape the Compose topology so translating it into a Helm chart later is mechanical (one Deployment+Service per container, one PVC for storage, one ConfigMap+Secret for env vars).

**Non-Goals:**
- The Helm chart itself — explicitly deferred to a future change once the container images/topology are proven locally.
- Production-grade orchestration concerns (replicas, resource limits, ingress/TLS, horizontal scaling) — Compose is a local dev tool, not a production target.
- CI image publishing/registry push — out of scope; this change only produces buildable Dockerfiles and a working local Compose stack.

## Decisions

### 1. Frontend served by `nginx:alpine` from a static build, not the Vite dev server
`Steward.Web`'s Dockerfile is multi-stage: a Node build stage (`npm ci && npm run build`) producing `dist/`, copied into an `nginx:alpine` final stage with a minimal `nginx.conf` (SPA fallback to `index.html`, gzip).
**Alternative considered**: run `npm run dev`/`vite preview` in the container — rejected, that's not what ships to production, and the whole point of this change is to exercise the real deployment shape locally.

### 2. Frontend's API base URL is injected at container *startup*, not baked in at build time
The built `dist/` bundle references a `window.__APP_CONFIG__.apiBaseUrl` (or equivalent) populated by a small `config.js` file generated by the Nginx container's entrypoint script from an `API_BASE_URL` environment variable, rather than a Vite build-time `import.meta.env.VITE_API_BASE_URL`.
**Rationale**: a build-time env var means a new image must be built per environment (dev/staging/prod), which directly fights the "build once, deploy anywhere" model Kubernetes/Helm expects (one image, promoted across environments with different ConfigMaps). Runtime injection keeps the same built image valid in Compose and later in any Helm-deployed environment.
**Alternative considered**: bake `VITE_API_BASE_URL` at build time — rejected for the reason above; acceptable to revisit only if runtime injection proves troublesome.

### 3. API configuration is overridden via Compose environment variables, not edits to `appsettings.Development.json`
`docker-compose.yml`'s `api` service sets `ConnectionStrings__DefaultConnection`, `Jwt__Key`, `Cors__AllowedOrigins__0`, `Frontend__BaseUrl`, `Storage__RootPath`, etc. (ASP.NET Core's double-underscore env var convention), sourced from a `.env` file (gitignored) with a checked-in `.env.example` documenting every required variable and a safe local default.
**Rationale**: this is exactly the ConfigMap/Secret split a Helm chart will need later — non-secret values become ConfigMap entries, secrets (JWT key, OAuth client secrets, DB password) become Secret entries — so getting in the habit of injecting via environment now is direct preparation, not extra work.

### 4. Storage volume: a named Docker volume locally, explicitly documented as the dev stand-in for the Kubernetes NFS-backed PVC
`docker-compose.yml` adds a `storage_data` named volume mounted at the path `Storage__RootPath` points to inside the `api` container (e.g. `/app/storage`). Named volumes (not host bind mounts) are used so behavior doesn't depend on host OS path/permission quirks (relevant given local dev happens on Windows).
**Alternative considered**: a host bind mount to a repo-relative folder (e.g. `./.storage`) — rejected as the default because of Windows bind-mount permission/performance inconsistencies; documented as an easy opt-in override for anyone who wants host-visible files, but the named volume is the default in the committed Compose file.

### 5. `api` waits on Postgres's healthcheck, not a fixed delay
`depends_on: postgres: condition: service_healthy` on the `api` service, reusing the existing Postgres healthcheck — avoids the classic "API starts before Postgres accepts connections" race without a sleep/retry hack inside the app.

## Risks / Trade-offs

- **[Risk]** Runtime-injected frontend config (Decision 2) is a small but nonstandard pattern relative to typical Vite tutorials → **Mitigation**: documented in the README; it's a well-known pattern for containerized SPAs (same approach used by most "12-factor SPA" guides) and pays for itself the moment a second environment exists.
- **[Risk]** Named volumes are less discoverable than a host folder when a developer wants to poke at uploaded files directly → **Mitigation**: README documents `docker compose exec api ls /app/storage` (or `docker volume inspect`) as the way to look inside; a host bind-mount override is documented as opt-in.
- **[Risk]** Drift between this Compose file and the eventual Helm chart if they're not kept in sync → **Mitigation**: explicitly out of scope to build the Helm chart now, but Decisions 3-4 are written specifically to make that future translation mechanical rather than a redesign.

## Migration Plan

No EF Core / database migration. This change only adds Dockerfiles, Compose services, an `.env.example`, and README documentation — no application or domain code changes. Rollback is simply not using the new Compose services (Postgres-only Compose usage continues to work unchanged).

## Open Questions

None — the Helm chart itself is explicitly deferred to a future change once this Compose topology is validated locally.
